项目名称：标准化秒杀商城系统

项目简介
基于 Spring Boot + MyBatis-Plus + Redis + RabbitMQ + Vue 开发的前后端分离电商秒杀系统，覆盖用户登录注册、商品展示、普通下单、秒杀下单、支付模拟、重复提交防护、令牌桶限流、消息队列异步落库等核心能力。项目重点解决了高并发秒杀场景下的接口限流、库存超卖、订单幂等和异步削峰问题，具备较强的业务完整性和技术展示度。

可写在简历里的项目描述
1. 设计并实现了秒杀与普通商品混合的电商下单体系，支持 JWT 登录鉴权、商品列表查询、秒杀抢购、模拟支付、订单查询等完整业务链路。
2. 基于 Redis Lua 脚本实现秒杀库存原子扣减和用户重复购买校验，结合 RabbitMQ 异步消费落库，解决高并发场景下的超卖与数据库瞬时压力问题。
3. 引入自定义令牌桶限流拦截器，对登录、秒杀等高频接口进行按用户/IP 维度的限流控制，提升系统稳定性并降低恶意刷接口风险。
4. 使用雪花算法生成全局唯一订单主键，支持分布式环境下的高并发写入，并将主订单与秒杀扩展订单分表记录，增强订单追踪与业务扩展能力。
5. 采用前后端分离架构，前端使用 Vue 构建登录、商品、秒杀首页与购买交互页面，后端通过统一响应体、全局异常处理和拦截器链路保证接口一致性与可维护性。

技术栈亮点与解释

一、Spring Boot + MyBatis-Plus：快速构建标准化后端
技术点说明：
- Spring Boot 负责项目启动、组件装配、REST 接口和拦截器管理。
- MyBatis-Plus 用于简化 CRUD、分页、条件查询和实体映射。

为什么值得写：
- 能体现你对企业级 Java Web 常用框架的掌握。
- 说明你不是只会写接口，而是能做完整的分层设计和持久层建模。

可以延伸的表达：
- 使用 DTO/VO/Entity 分层，避免前后端字段耦合。
- 使用 Mapper + Service + Controller 的标准架构提升代码可读性。


二、Redis + Lua：秒杀库存原子扣减
技术点说明：
- 秒杀库存先放在 Redis 中做热点访问。
- 通过 Lua 脚本保证“判断是否购买过 + 判断库存是否充足 + 扣减库存”这几个动作原子执行。

解决的问题：
- 避免并发下库存被多次扣减导致超卖。
- 避免用户重复抢购同一商品。
- 提升秒杀接口响应速度，减少数据库压力。

简历表达建议：
- “基于 Redis Lua 实现秒杀库存原子扣减，支持防超卖与重复下单校验。”

可延伸解释：
- Lua 脚本在 Redis 中单线程执行，天然保证原子性。
- 适合做库存、验证码、幂等标记等高并发热点数据处理。


三、RabbitMQ：异步削峰与最终一致性
技术点说明：
- 秒杀请求扣库存成功后，不直接写库，而是发送 MQ 消息。
- 消费者异步创建普通订单和秒杀扩展订单。

解决的问题：
- 削峰填谷，避免高并发请求直接冲击数据库。
- 主流程快速返回，提高用户体验。
- 通过消息消费实现“下单成功后异步落库”。

简历表达建议：
- “使用 RabbitMQ 将秒杀订单创建流程异步化，缓解数据库瞬时写入压力并提升系统吞吐量。”

可延伸解释：
- MQ 适合解耦主流程和耗时操作。
- 通过消费者幂等和事务控制，保证消息重复投递时不会重复落库。


四、幂等设计：唯一键 + 事务回滚
技术点说明：
- 秒杀订单表通过唯一索引约束同一用户、同一活动、同一商品只能存在一条记录。
- 消费者在插入订单时先查重，插入失败时回滚事务，避免脏数据。

解决的问题：
- 处理重复消息、网络重试、并发消费导致的重复订单问题。
- 防止“普通订单插入成功，但秒杀扩展表失败”的孤儿数据。

简历表达建议：
- “设计秒杀订单幂等机制，结合数据库唯一约束与事务回滚控制重复下单问题。”

可延伸解释：
- 幂等是高并发和分布式系统的核心问题之一。
- 常见实现方式包括唯一键、幂等 token、状态机、去重表等。


五、令牌桶限流：自定义拦截器 + 注解化配置
技术点说明：
- 自定义 `@TokenBucketRateLimit` 注解标记接口。
- `TokenBucketRateLimitInterceptor` 在请求进入 Controller 前执行限流判断。
- 通过 Redis + Lua 实现令牌桶算法，支持按用户 ID 或 IP 维度限流。

解决的问题：
- 防止登录接口、秒杀接口被高频刷请求。
- 保护系统核心资源，降低恶意流量影响。

简历表达建议：
- “基于自定义注解和 Redis Lua 实现令牌桶限流，对高频接口按用户/IP 粒度进行访问控制。”

可延伸解释：
- 令牌桶相比固定窗口更平滑，能允许突发流量但限制长期超频。
- 注解化配置让限流策略可复用、可维护。


六、JWT + 登录态上下文：无状态鉴权
技术点说明：
- 登录成功后生成 JWT，前端保存 token，后续请求携带 token。
- 后端通过拦截器解析 token，并将登录用户放入 `LoginUserContext`。

解决的问题：
- 免去服务端 session 存储，适合前后端分离。
- 在限流、下单、支付等逻辑中可快速获取当前登录用户。

简历表达建议：
- “采用 JWT 实现无状态登录鉴权，结合拦截器上下文统一管理用户身份信息。”

可延伸解释：
- 无状态认证便于水平扩展。
- `LoginUserContext` 让业务层无需重复解析 token。


七、雪花算法：分布式唯一 ID 生成
技术点说明：
- 使用雪花算法生成订单主键 `id`。
- 订单号 `orderNo` 由业务时间戳 + 用户信息 + 雪花 ID 组合生成，兼顾可读性和唯一性。

解决的问题：
- 避免数据库自增 ID 在分库分表或高并发场景下的局限。
- 保证多实例部署时主键唯一。

简历表达建议：
- “使用雪花算法生成订单主键，支持分布式场景下的高并发唯一 ID 写入。”

可延伸解释：
- 雪花 ID 一般由时间戳、机器位、序列号组成。
- 具有趋势递增特性，适合索引写入。


八、前后端分离与统一异常处理
技术点说明：
- 前端使用 Vue 组织登录、商品、秒杀首页。
- 后端通过统一 `ResponseResult` 返回结果。
- 通过全局异常处理器统一处理业务异常和系统异常。

解决的问题：
- 提升接口返回一致性。
- 方便前端统一处理成功/失败场景。
- 提高调试效率和可维护性。

简历表达建议：
- “采用前后端分离架构与统一响应体设计，配合全局异常处理提升接口一致性和可维护性。”

可延伸解释：
- 统一返回结构便于前端状态管理。
- 全局异常处理避免 Controller 中大量 try-catch。


九、数据库建模：普通订单 + 秒杀订单双表设计
技术点说明：
- `t_order` 存储主订单通用信息。
- `t_seckill_order` 存储秒杀扩展信息，例如活动 ID、秒杀价快照、购买数量、优惠金额。

解决的问题：
- 普通订单和秒杀订单业务隔离。
- 历史订单数据保留快照，避免商品信息变更影响订单展示。

简历表达建议：
- “采用主订单 + 秒杀扩展表的双表设计，实现订单快照、业务隔离与后续扩展能力。”

可延伸解释：
- 这种设计适合复杂业务中的“主表 + 扩展表”模式。
- 有利于兼容未来优惠券、拼团、预售等扩展场景。


项目亮点总结（简历可直接使用）
- 设计了秒杀库存 Redis 原子扣减方案，结合 Lua 脚本和 MQ 异步落库，实现防超卖、抗高并发和最终一致性。
- 实现了注解化令牌桶限流，支持登录和秒杀等核心接口的用户/IP 维度访问控制。
- 引入雪花算法生成分布式唯一订单 ID，并结合业务订单号提高系统可追踪性。
- 采用 JWT、全局异常处理、统一响应体、前后端分离等工程化手段，提升系统安全性、稳定性和可维护性。
